# Revenge Of The Cute Magical Router Gateway writeup

## Задание

Дан таск `Revenge of the Cute Magical Router Gateway` с таким описанием:

```text
Yet another router was found in the backrooms of THEM hq.
...
I strongly feel that the password might contain the string "them"
...
hint 1: ... rock is really popular in the password cracking scene. Maybe you should use rock too!
hint 2: I randomly tried a rlly long username, and somehow it took a lot longer for the router to respond.
hint 3: The Username and Password are both in the famous rockyou database leak.
```

Из условия нам важны четыре вещи:

- и `username`, и `password` есть в `rockyou.txt`
- пароль, скорее всего, содержит `them`
- время ответа зависит от того, что мы отправляем
- это не совсем обычный web exploitation

## Основная идея

Это **timing side-channel на логине**.

Подсказки прямо ведут к этому:

- `rock` явно намекает на `rockyou`
- странное замедление на длинном username означает, что сервер, вероятно, сравнивает строки не constant-time
- раз оба секрета лежат в `rockyou`, задача превращается из полного brute force в словарный поиск с оракулом по времени

То есть вместо "взлома вебки" мы используем сервис как измеримый оракул:

```text
отправить кандидат -> измерить задержку -> понять, насколько далеко дошла проверка
```

## Что именно значит подсказка про длинный username

Вторая подсказка фактически сливает уязвимость.

Если длинный username заставляет роутер отвечать заметно дольше, значит backend, скорее всего, использует сравнение, время которого зависит от длины строки или длины совпавшего префикса.

Типичные плохие варианты:

```c
strcmp(user_input, real_username)
strncmp(user_input, real_username, strlen(user_input))
memcmp(...)
for (...) if (a[i] != b[i]) return fail;
```

В таком случае латентность начинает зависеть от:

- длины отправленной строки
- количества совпавших символов в префиксе
- того, выполняется ли проверка username раньше password

Этого уже достаточно, чтобы превратить логин в источник утечки.

## Почему `rockyou` решает половину задачи

Без словаря тайминг всё равно был бы полезен, но пространство вариантов было бы намного больше.

С `rockyou.txt` всё резко упрощается:

- правильный `username` точно есть в `rockyou`
- правильный `password` тоже точно есть в `rockyou`

Значит, реальное решение разбивается на два этапа:

1. найти `username` по таймингу
2. зафиксировать найденный `username`
3. найти `password` по таймингу

Это больше похоже на side-channel-assisted dictionary cracking, чем на обычную веб-уязвимость.

## Этап 1. Восстановить username

Самый удобный путь:

- берем кандидат из `rockyou`
- подставляем заведомо неверный пароль
- много раз меряем время ответа

Если сервер сначала проверяет username, а потом уже password, то правильный username должен в среднем отвечать дольше, потому что код успевает пройти на один шаг глубже перед отказом.

Практически я бы делал так:

1. убрать дубликаты из `rockyou`
2. в первом проходе прогнать крупные батчи
3. каждый кандидат проверять несколько раз
4. ранжировать по `median latency`, а не по одному измерению

Псевдокод:

```python
for username in rockyou:
    timings = []
    for _ in range(N):
        t0 = now()
        login(username, "definitely_wrong_password")
        timings.append(now() - t0)
    score[username] = median(timings)

best_usernames = top_k(score)
```

Обычно после такого первого прохода остается маленький shortlist.

## Этап 2. Уточнение через префиксы

Если реализация течет посимвольно, то можно идти еще эффективнее.

Вместо того чтобы тестировать каждое слово независимо, полезно группировать кандидатов по префиксам и сравнивать медианы по группам.

Например:

- все username на `a`
- все на `b`
- ...

Потом рекурсивно:

- `th`
- `the`
- `them`

То есть тайминг превращается в поиск по trie.

А раз username точно берется из `rockyou`, то такой prefix-based filtering работает особенно хорошо.

## Этап 3. Восстановить password

Когда корректный username уже найден, фиксируем его и повторяем ту же процедуру для пароля.

Теперь сигнал обычно чище, потому что сервер уже не отваливается на первой же проверке username.

Здесь особенно полезна подсказка:

```text
the password might contain the string "them"
```

Поэтому на первом проходе имеет смысл приоритизировать:

- все пароли из `rockyou`, содержащие `them`
- а уже потом остальные

Это резко уменьшает число запросов.

Псевдокод:

```python
for password in prioritized_rockyou:
    timings = []
    for _ in range(N):
        t0 = now()
        login(real_username, password)
        timings.append(now() - t0)
    score[password] = median(timings)
```

Лучший кандидат - тот, который стабильно дает максимальное время ответа или сразу переводит сервис в success path.

## Практические замечания

В реальной атаке важны детали:

- делать несколько измерений на один кандидат
- использовать `median`, а не обычное среднее
- держать стабильный ритм запросов
- устраивать второй проход по лучшим кандидатам
- не пытаться делать выводы по одному-двум ответам

Для удаленного инстанса тайминг почти всегда шумный.
Здесь intended solve именно статистический.

## Почему это не совсем web

Условие честно предупреждает об этом.

Веб-интерфейс здесь лишь удобный frontend к оракулу.
Суть уязвимости в другом:

- secret-dependent comparison time
- плюс маленькое словарное пространство поиска

Поэтому по духу это ближе к:

- side-channel analysis
- dictionary cracking

а не к классическим web-багам вроде SQLi, XSS или auth bypass.

## Главная мысль

Урок задачи очень простой:

- тайминговая утечка становится крайне опасной, если секрет выбирается из небольшого словаря
- последовательная проверка `username`, потом `password` делает утечку сильнее
- даже маленькая подсказка вроде `contains "them"` драматически сужает поиск

Это хороший пример того, как вроде бы "слабый" timing leak становится полностью практичным, если рядом лежит `rockyou`.

## Примечание

Конкретную пару `username/password` можно восстановить только через работу с живым инстансом и замеры времени ответа.
