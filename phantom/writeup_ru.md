# phantom writeup

## Задание

Дан сервис `phantom`.
На вход он принимает hex-encoded байткод для собственной VM.

Формат флага:

```text
THEM?!CTF{...}
```

## Первичный анализ

По строкам сразу видно, что это не обычный menu-pwn, а кастомная виртуальная машина:

```text
"The dead speak in bytecode..."
Submit your phantom script as hex-encoded bytecode.
Max code size: 2048 bytes (4096 hex chars)
```

Также были полезны строки ошибок:

```text
invalid register
truncated PUSH
truncated POP
truncated JMP
truncated JZ
truncated JNZ
truncated PUSHR
truncated MOV
truncated INC
truncated PEEK
truncated POKE
unknown opcode
```

По `checksec`:

- `NX enabled`
- `PIE disabled`
- бинарь `statically linked`
- бинарь `stripped`

Это удобно для эксплуатации:

- код по фиксированным адресам
- можно строить `ROP` прямо по самому бинарю

## Поведение программы

Программа печатает баннер, просит hex-строку, декодирует ее в байткод и затем исполняет:

```text
phantom> [*] Loaded %zd bytes of bytecode. Executing...
```

Максимальный размер байткода после декодирования: `2048` байт.

## Разбор VM

Основной интерпретатор находится в функции по адресу `0x4019ee`.

После разбора опкодов получаем примерно такую картину:

- `0x01` - `HALT`
- `0x02` - `PUSH imm64`
- `0x03` - `POP reg`
- `0x04` - `ADD`
- `0x05` - `SUB`
- `0x06..0x0b` - арифметика и битовые операции
- `0x0c` - сравнение, результат уходит во внутренний флаг
- `0x0d` - `JMP imm16`
- `0x0e` - `JZ imm16`
- `0x0f` - `JNZ imm16`
- `0x10` - печать вершины стека
- `0x11` - `DUP`
- `0x12` - `SWAP`
- `0x13` - `PUSHR reg`
- `0x14` - `INC reg`
- `0x15` - `PEEK reg`
- `0x16` - `POKE reg`
- `0x17` - `MOV dst, src`

Внутреннее состояние VM лежит прямо в локальном стеке функции интерпретатора:

- область регистров
- стек VM
- счетчики `ip/sp`
- сам буфер байткода

Это сразу наводит на мысль, что `PEEK/POKE` надо смотреть особенно внимательно.

## Уязвимость

Критичный баг в том, что `PEEK` и `POKE` используют значение регистра как индекс в массиве стековых слотов VM, но **не проверяют границы** этого индекса.

Упрощенно логика выглядит так:

```c
value = frame[reg[idx] + CONST];      // PEEK
frame[reg[idx] + CONST] = value;      // POKE
```

Проверяется только:

- валиден ли номер регистра `0..15`
- достаточно ли элементов в VM-стеке для самой операции

Но **не проверяется**, что индекс из регистра попадает внутрь разрешенной области.

Итог:

- `PEEK` дает out-of-bounds read по стековому фрейму интерпретатора
- `POKE` дает out-of-bounds write по стековому фрейму интерпретатора

## Почему это приводит к RCE

Поскольку интерпретатор работает в обычной функции, в его стековом фрейме рядом лежат:

- сохраненный `RBP`
- сохраненный `RIP`
- локальные переменные
- данные байткода

Через OOB-доступ можно:

1. прочитать сохраненный `RBP`
2. вычислить адреса нужных областей на стеке
3. перезаписать `saved RIP`
4. разложить после него `ROP`-цепочку

То есть задача сводится к обычному `stack-based ROP`, только write primitive получаем не через overflow, а через баг VM.

## Нужные смещения

Во время отладки выяснилось, что относительно индексируемой области VM:

- `saved rbp` находится по индексу `0x4e`
- `saved rip` находится по индексу `0x4f`

Этого достаточно:

- через `PEEK 0x4e` можно получить текущую стековую базу
- через `POKE 0x4f` и дальше можно записать `ROP`

## Идея эксплуатации

Эксплуатация состояла из 3 шагов.

### 1. Получить адреса на стеке

Сначала байткод:

- читает `saved rbp`
- вычисляет базу текущего стекового фрейма
- считает абсолютные адреса:
  - строки `/home/flag.txt`
  - буфера под `read`

### 2. Записать полезные данные на стек

Через `POKE` в контролируемые слоты кладутся:

- строка `/home/flag.txt\x00`
- буфер под чтение файла
- `ROP`-цепочка

### 3. Перехватить возврат из интерпретатора

Поскольку `saved RIP` перезаписан, после завершения VM управление уходит в `ROP`.

Цепочка делает:

1. `open("/home/flag.txt", 0, 0)`
2. `read(3, buf, 0x100)`
3. `write(1, buf, 0x100)`
4. `exit(0)`

## ROP

Так как бинарь не PIE и статически слинкован, гаджеты брались прямо из `phantom`.

Использовались, например:

- `pop rdi ; ret`
- `pop rsi ; ret`
- `pop rdx ; pop rbx ; ret`
- `pop rax ; ret`
- `syscall ; ret`

Этого достаточно для syscall-based ROP.

## Проверка локально

Перед удаленной проверкой эксплойт был протестирован на локальном файле, например `/etc/hostname`.

После корректной подстройки смещений байткод успешно читал файл и печатал его содержимое.

Это подтвердило, что:

- OOB read/write работает
- `saved RIP` контролируется
- `ROP` исполняется корректно
