# Entropy Core writeup

## Задание

Дан Windows-бинарь `entropy_core.exe`.

По строкам видно, что программа печатает приглашение:

```text
[Entropy Core v1] quantum lattice primed -- insert star key:
```

а затем либо:

```text
[Entropy Core] starlight conduit aligned. access accepted.
```

либо:

```text
[Entropy Core] CRITICAL: warp drift -- reactor SCRAM.
```

Формат флага:

```text
THEM?!CTF{...}
```

## Первые наблюдения

Это не обычный crackme с прямым сравнением введенной строки.

Внутри бинаря сидит кастомная VM:

- 16 регистров
- 64-битная арифметика
- диспетчер опкодов через jump-table
- встроенная bytecode-программа в `.rdata`

Строка:

```text
EntropyCoreV1!
```

легко уводит в сторону.
В бинаре действительно есть фейковая структура, похожая на RC4-логику, но основная задача вообще не в этом.

Настоящая логика проверки находится внутри VM.

## Как устроена проверка

После разбора диспетчера становится понятно, что программа:

1. читает кандидат на флаг,
2. прогоняет его посимвольно через интерпретатор байткода,
3. обновляет состояние VM из 16 регистров,
4. сравнивает младший байт вычисленного значения с 36-байтным target, зашитым в самой bytecode-программе.

То есть по сути это кастомный verifier:

- каждый байт ввода влияет на состояние VM
- байткод преобразует это состояние
- результат должен совпасть с захардкоженной последовательностью

## Две главные ловушки

Задача неприятна в основном из-за двух моментов.

### 1. Фальшивый криптографический след

В бинаре есть RC4-подобные данные и строка `EntropyCoreV1!`, из-за чего легко начать разбирать всё как обычную stream-cipher схему.

Это ложный след.

Реальная валидация сидит в VM, а не в стандартной криптографии.

### 2. Сдвиги используют immediate, а не регистр

Самый важный нюанс при реверсе: инструкции вида

- `ROL`
- `ROR`
- `SHL`
- `SHR`

берут величину сдвига **не из регистра**.

Количество бит для сдвига хранится прямо в байткоде как третий байт операнда.

Если ошибочно трактовать этот байт как индекс регистра, вся семантика VM съезжает, и любой solver начинает давать мусор.

Это как раз тот маленький баг в реверсе, из-за которого таск кажется нерешаемым, пока не поправишь декодирование операндов.

## Стратегия решения

После того как семантика VM восстановлена правильно, проверка становится детерминированной.

Практический путь решения такой:

1. разобрать доступные опкоды,
2. заэмулировать встроенную bytecode-программу,
3. понять, как каждый входной байт меняет состояние VM,
4. сверять вычисленный байт с target для соответствующей позиции,
5. восстанавливать флаг посимвольно.

Наивный greedy-подход здесь плох:

- на ранних позициях может быть несколько printable-кандидатов
- первый подходящий байт не гарантирует, что дальше путь останется валидным

Поэтому надежный вариант - DFS с backtracking по printable ASCII.

Этого достаточно, потому что:

- длина флага фиксирована логикой проверки
- каждое следующее состояние сильно ограничивает продолжение
- неверные ветки быстро отваливаются

## Полученный флаг

После исправления трактовки shift-операндов и запуска backtracking solver по printable ASCII получается единственный валидный ввод:

```text
THEM?!CTF{Entr0py_C0r3_VM_S0_Funny!}
```
