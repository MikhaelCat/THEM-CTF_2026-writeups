# Ancient Signals writeup

## Задание

Дан таск `Ancient Signals` на `100` очков со следующим описанием:

```text
Our field agents recovered a secure comms package from an enemy operative. They built a makeshift tool to translate the weird signals into an audible frequency, but the extraction corrupted the software. Can you help us in fixing this software so we can hear what's on the tape?
```

В архиве были два файла:

- `player.exe`
- `transmission.dat`

Формат флага:

```text
THEM?!CTF{...}
```

## Первичный анализ

Описание специально подталкивает нас к идее "починить" софт и прослушать запись.
Но это лишь часть правды.

Здесь есть две отдельные сущности:

- зашифрованная запись в `transmission.dat`
- Windows-бинарь `player.exe`

На первый взгляд кажется, что путь решения такой:

1. починить плеер,
2. расшифровать запись,
3. воспроизвести звук,
4. достать флаг из аудио.

На деле это мисдирекшн: аудио действительно восстанавливается, но **настоящий флаг лежит внутри самого `player.exe`**.

## Разбор `transmission.dat`

Файл с записью не похож на обычный аудиоконтейнер.
Он не начинается с `RIFF` и выглядит как зашифрованный бинарный поток.

По структуре байтов видно, что это:

- 8-bit unsigned PCM
- XOR с повторяющимся ключом длиной 128 байт

Для `8-bit unsigned PCM` тишина кодируется байтом `0x80`.

Это сразу дает удобный known-plaintext:

```python
key[i % 128] = data[i] ^ 0x80
```

если найти участок, который после расшифровки должен быть тишиной.

Такой участок в файле есть.
Блок примерно `0x50..0xCF` затем повторяется еще раз, что очень похоже на постоянный участок сигнала, то есть на тишину.

Дальше решение простое:

1. восстановить 128-байтный XOR-ключ по "тихому" участку,
2. расшифровать весь файл,
3. сохранить результат как wave.

После XOR файл начинаетcя с:

```text
RIFF .... WAVE
```

То есть запись действительно является корректным аудио.

## Почему запись не дает флаг

После расшифровки оказывается, что внутри обычный рикролл.

То есть даже если успешно "починить программу" и получить звук, флаг мы все равно не найдем.

В этом и состоит основной трюк задачи:

- запись действительно решается,
- аудио настоящее,
- но это не тот секрет, который нам нужен.

## Разбор `player.exe`

Реальный флаг извлекается из Windows-бинаря.

При просмотре `.data` можно заметить небольшой зашифрованный blob.
В открытом виде он не хранится и напрямую не связан с выводом аудио.

Дальнейший реверс показывает, что плеер строит XOR-ключ из собственного кода:

- берет небольшой кусок `.text`
- считает от него `FNV-1a`
- использует байты хэша как XOR-ключ для скрытого blob

Ключевой кусок кода находится в небольшой anti-tamper функции, которая проверяет, что вход начинается с `"RIFF"`.

Соответственно, реальный путь такой:

1. найти зашифрованный blob в `.data`,
2. определить байты `.text`, которые идут в хэш,
3. посчитать `FNV-1a` от этого среза кода,
4. расшифровать blob через XOR полученным ключом.

## Почему не нужно реально чинить плеер

В условии сказано, что софт поврежден и надо сделать запись слышимой.
Но сам плеер не является основной проблемой.

В бинаре есть проверка на то, что сырые входные данные начинаются с `RIFF`.
Зашифрованный `transmission.dat` этому условию, очевидно, не соответствует, поэтому программа не приходит к полезной ветке исполнения.

Из-за этого попытка именно "допилить аудиоплеер" только тратит время.

Самый короткий путь - реверснуть бинарь напрямую и достать скрытый blob.

## Получение флага

После вычисления правильного `FNV-1a` и XOR-расшифровки скрытых байтов получаем:

```text
THEM?!CTF{1mag1n3_gett1ng_r1ckr0ll3d_1n_tH3M?!C7F_xDDD}
```
