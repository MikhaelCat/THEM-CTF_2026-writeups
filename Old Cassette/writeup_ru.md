# Old Cassette writeup

## Задание

Дан таск `Old Cassette`.

В архиве лежит один файл:

- `main.bin`

Формат флага:

```text
THEM?!CTF{...}
```

## Первый взгляд

Размер файла всего около `3.2 KB`, что уже намекает, что перед нами не обычный исполняемый файл, а ROM для какой-то маленькой VM или консоли.

Быстрый hex dump сразу выдает формат:

```text
00E0 1280
```

Для CHIP-8 это означает:

- `00E0` - очистить экран
- `1280` - перейти на `0x280`

То есть задача представляет собой **CHIP-8 ROM**.

Из этого сразу следуют полезные выводы:

- код начинается с адреса `0x200`
- графика строится простыми монохромными спрайтами
- здесь не будет OS-специфичных трюков
- решать задачу нужно через разбор логики ROM

## Общая структура ROM

После дизассемблирования у ROM быстро выделяются ключевые части:

- `0x2c0` - основная функция перехода состояния
- `0x322` - диспетчер, выбирающий одну из таблиц данных
- `0x900` - главная рутина, которая проводит все раунды и рисует символы
- `0xdd2` - helper для отрисовки символов на экран

Важно, что программа вообще не просит никакого ввода.
Ее единственная задача - **посчитать символы и нарисовать их на дисплее**.

Значит, вместо "запуска ROM как игры" нам нужно ответить лишь на два вопроса:

1. как вычисляется каждый символ?
2. куда именно этот символ рисуется?

## Главная рутина по адресу `0x900`

Код на `0x900` очищает экран, инициализирует двухбайтное состояние, а затем 32 раза повторяет одну и ту же схему.

Начальное состояние:

```text
VA = 0xA7
VB = 0xC3
```

Каждый раунд делает три вещи:

1. много раз продвигает внутреннее состояние,
2. по новому состоянию вычисляет один печатный символ,
3. рисует его в позиции `(x, y)`.

Повторяющиеся блоки легко видны по шаблону:

```text
задать счетчик
вызвать 0x282 / 0x2ac
вычислить байт через таблицы
вызвать 0xdd2 для отрисовки
```

Первые 16 раундов используют явные счетчики:

```text
1, 4, 16, 64, 256, 1024, ...
```

То есть это просто степени четверки.

Последние 16 раундов идут через обертку, которая многократно вызывает ту же самую функцию с заполненными `0xff` байтами счетчика.
Если читать это в лоб, выглядит страшно.

Именно на это задача и рассчитана.

## Машина состояний в `0x2c0`

Подпрограмма на `0x2c0` преобразует 16-битное состояние `(VA, VB)` в новое `(VA', VB')`.

Если смотреть на высоком уровне, она делает следующее:

1. копирует текущее состояние в рабочие регистры,
2. читает байт из таблицы по адресу `0x800 + VB`,
3. XOR'ит этот байт с `VB`,
4. еще раз XOR'ит с константой, зависящей от двух старших битов `VB`,
5. добавляет результат в `VB`,
6. пять раз перемешивает промежуточное 16-битное значение сдвигами и переносами,
7. получает новое `(VA, VB)`,
8. сохраняет его в `0x58b` и `0x58c`.

Критически важно здесь не конкретное выражение, а то, что эта функция является **чистой функцией от 16-битного состояния**.

А значит, всего существует лишь `2^16 = 65536` возможных состояний.
При многократном применении такая функция обязательно рано или поздно войдет в цикл.

То есть сколько бы огромными ни выглядели счетчики, бесконечно новые состояния она генерировать не может.

## Почему brute force здесь неправильный путь

Если буквально эмулировать все шаги, поздние раунды огромны.

Обертка на `0x2ac` фактически прогоняет:

```text
255 * (2^32 - 1)
```

переходов состояния на один раунд.

В лоб такое эмулировать бессмысленно.

Но как только мы замечаем, что пространство состояний маленькое, правильная стратегия становится очевидной:

1. итерируем функцию из стартового состояния `(0xA7, 0xC3)`,
2. запоминаем, когда каждое состояние встретилось впервые,
3. находим хвост и цикл,
4. отвечаем на `state_after(N)` за O(1).

При таком разборе траектория получилась такой:

- длина хвоста: `329`
- длина цикла: `34`

То есть уже через несколько сотен шагов все дальнейшее поведение становится периодическим.

Это полностью ломает задуманный "экспоненциальный" ужас.

## Как восстановить `state_after(N)`

Пусть у нас есть:

- `prefix` - докцикловая часть,
- `cycle` - повторяющийся кусок.

Тогда:

```python
def state_after(n):
    if n < tail_len:
        return states[n]
    return states[tail_len + ((n - tail_len) % cycle_len)]
```

После этого даже чудовищно большие счетчики становятся тривиальными.

## Как вычисляется один символ

После продвижения состояния ROM получает код символа из нового `(VA, VB)`.

Ключевой кусок выглядит так:

```text
V0 = VA
V1 = 7
V0 &= 7
call 0x322
V1 = offset
I += V1
V0 = mem[I]
V9 = V0
V9 ^= VA
V9 ^= VB
```

То есть символ вычисляется по формуле:

```text
char = mem[BASE[VA & 7] + offset] ^ VA ^ VB
```

где `BASE[...]` выбирается диспетчером по адресу `0x322`.

Этот диспетчер мапит значения так:

```text
0 -> 0x400
1 -> 0x460
2 -> 0x4c0
3 -> 0x520
4 -> 0x600
5 -> 0x660
6 -> 0x6c0
7 -> 0x720
```

У каждого раунда при этом зашиты:

- свой `offset`
- свои координаты `(x, y)`

Так что после восстановления состояния все остальное уже механика.

## Извлечение всех 32 символов

В ROM есть 32 одинаково устроенных блока отрисовки.

Для каждого из них я вытащил:

- число итераций,
- смещение внутри выбранной таблицы,
- координаты на экране.

После этого solve-логика выглядит так:

1. стартуем из `(VA, VB) = (0xA7, 0xC3)`,
2. продвигаем состояние на счетчик текущего раунда,
3. считаем байт символа,
4. кладем его в найденную позицию `(x, y)`,
5. сортируем символы по строкам и слева направо.

На экране получается:

```text
THEM?!CTF{
0LD_T4P3_N
3V3R_D1E5K
7}
```

THEM?!CTF{0LD_T4P3_N3V3R_D1E5K7}

## Вывод

Задача выглядит жутко, потому что маленькую машину состояний специально прячут за очень большими счетчиками.

Но главный урок здесь простой:

- если преобразование детерминировано,
- если пространство состояний маленькое,
- то задача сводится не к "долгой эмуляции", а к поиску цикла.

Как только функция перехода восстановлена правильно, все остальное - это просто извлечение координат и смещений из главной рутины.
