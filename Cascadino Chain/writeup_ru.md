# Cascadino Chain writeup

## Задание

Дан crypto-таск `Cascading Chain` на `100` очков с описанием:

```text
I built an unbreakable encryption chain, each key protects the next and the last loops back to the start. There's no way in... right?
```

Также даны четыре hex-строки:

```text
c1 = 36273f225d4e393b2414025f1030025f1030025f10301907035e145e0c082508520a0930001d081d1012
c2 = 0e021b000e021b000e021b000e021b000e021b000e021b000e021b000e021b000e021b000e021b000e02
c3 = 180504021805040218050402180504021805040218050402180504021805040218050402180504021805
c4 = 202020204b49263932131d5d06371d5d06371d5d0637060515590b5c1a0f3a0a440d1632161a171f0615
```

Формат флага:

```text
THEM?!CTF{...}
```

## Первые наблюдения

Все четыре строки имеют одинаковую длину.
Это сразу очень похоже на XOR, а не на блочные шифры или хэши.

Текст условия тоже прямо подсказывает структуру:

- один ключ защищает следующий
- последний замыкается обратно на начало

Естественная модель такая:

```text
c1 = flag ^ k1
c2 = k1   ^ k2
c3 = k2   ^ k3
c4 = k3   ^ flag
```

## Главный инвариант

Если XOR'нуть все четыре уравнения, то каждый секрет встречается два раза:

```text
c1 ^ c2 ^ c3 ^ c4
= (flag ^ k1) ^ (k1 ^ k2) ^ (k2 ^ k3) ^ (k3 ^ flag)
= 0
```

Это легко проверяется:

```python
assert bytes(a ^ b ^ c ^ d for a, b, c, d in zip(c1, c2, c3, c4)) == b"\x00" * len(c1)
```

То есть цепочка действительно корректна.

Но здесь же видно и основную проблему:

- только по `c1..c4` система недоопределена
- существует бесконечно много допустимых `(flag, k1, k2, k3)`

Значит, одной алгебры недостаточно.
Нужен внешний constraint.

## Внешний constraint: формат флага

Мы знаем, что флаг начинается с:

```text
THEM
```

Из первого уравнения:

```text
c1 = flag ^ k1
```

получаем:

```text
k1[0:4] = c1[0:4] ^ b"THEM"
```

Считаем:

```python
>>> bytes(a ^ b for a, b in zip(c1[:4], b"THEM"))
b'bozo'
```

Это уже очень сильная подсказка.
Первые четыре байта ключа выглядят не как случайный мусор, а как нормальное слово:

```text
bozo
```

При этом `c2` и `c3` визуально явно периодичны с периодом 4.
Значит, логично попробовать, что и `k1` - это 4-байтный повторяющийся ключ:

```text
k1 = b"bozo" repeated
```

## Восстановление флага

Тогда первое уравнение сразу дает:

```python
flag = c1 ^ (b"bozo" * 11)
```

Проверка дает:

```text
THEM?!CTF{x0r_x0r_x0r_cha1n1ng_g0es_brrrr}
```

## Проверка остальных ключей

Чтобы убедиться, что это не совпадение, можно восстановить и остальные ключи:

```text
k2 = k1 ^ c2 = "lmao"
k3 = k2 ^ c3 = "them"
```

После этого финальная проверка:

```text
k3 ^ flag == c4
```

тоже полностью сходится.

## Solve script

```python
c1 = bytes.fromhex("36273f225d4e393b2414025f1030025f1030025f10301907035e145e0c082508520a0930001d081d1012")

k1 = b"bozo" * 11
flag = bytes(a ^ b for a, b in zip(c1, k1))

print(flag.decode())
```

## Почему схема ломается

Заявление автора одновременно и верно, и неверно:

- если бы у нас не было вообще никакой дополнительной информации, замкнутая XOR-цепочка действительно не раскрывала бы единственный plaintext
- но в CTF формат флага дает очень сильный known-plaintext crib

Как только известны первые байты `THEM`, первый ключ начинает течь сразу.
А как только в чистой XOR-цепочке утек один ключ, вся конструкция разваливается полностью.

## Итоговый флаг

```text
THEM?!CTF{x0r_x0r_x0r_cha1n1ng_g0es_brrrr}
```
