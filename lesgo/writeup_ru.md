# checker writeup

## Задание

Дан бинарь `checker`.
Формат флага: `THEM?!CTF{...}`.

## Первичный анализ

Сначала по `file` было видно, что это:

- `PE32+ executable`
- `x86-64`
- Windows console binary

По строкам и служебным символам быстро стало понятно, что бинарь собран на Go:

- много `runtime.*`
- `fmt.*`
- имена вида `main.(*nodeA).Check`

Это было важное наблюдение, потому что логика проверки не была сильно заобфусцирована, а была разложена по нескольким маленьким Go-функциям.

## Поведение программы

При запуске через `wine` выясняется, что флаг нужно передавать аргументом командной строки:

```text
Usage: ./chall '<flag>'
```

Неверный ввод дает:

```text
Wrong!
```

## Полезные функции

Из Go-метаданных и `pclntab` были найдены реальные функции:

- `main.(*nodeA).Check`
- `main.(*nodeB).Check`
- `main.(*nodeC).Check`
- `main.(*nodeD).Check`
- `main.(*nodeE).Check`
- `main.buildMaze`
- `main.main`

Дальше задача свелась к восстановлению этих `Check`-функций.

## Структура проверки

`main.main` выделяет буфер на `0x28 = 40` байт и копирует туда `min(len(arg), 40)` байт входа.

После этого строится набор из 5 узлов, каждый из которых проверяет блок по 8 байт.

То есть флаг проверяется как 5 независимых кусков:

```text
[0:8] [8:16] [16:24] [24:32] [32:40]
```

## Node A

Первая функция использует два 8-байтных immediate-значения и сравнивает:

```text
input[i] XOR const1[i] == const2[i]
```

Отсюда восстанавливается первый блок:

```text
THEM?!CT
```

## Node B

Вторая функция берет таблицу из `.rdata` по адресу, соответствующему строке `F{n0rm4L`, и делает проверку после `rol 3`.

Так как поворот на 3 бита применяется и к входу, и к константе одинаково, блок сразу читается как:

```text
F{n0rm4L
```

## Node C

Третья функция использует таблицу `_g0r0ut1`.

Проверка состоит из двух частей:

1. сумма всех байтов входа должна совпасть с суммой байтов константы
2. XOR байтов входа должен совпасть с XOR байтов константы с учетом первого символа

Этого достаточно, чтобы подтвердить третий блок:

```text
_g0r0ut1
```

## Node D

Это место сначала оказалось самым неприятным.

Функция использует 8 байт:

```text
b0 a1 92 d5 d7 cd 9d 95
```

Сначала можно ошибочно решить, что там накопительная сумма, и тогда получается мусор.
Но правильная логика такая:

- стартовое значение: `0x42`
- на каждом шаге сравнивается `input[i] + previous_input`
- потом `previous_input` становится текущим `input[i]`

То есть:

```text
x0 + 0x42 = 0xb0
x1 + x0   = 0xa1
x2 + x1   = 0x92
...
```

После корректного восстановления получается:

```text
n3_val1d
```

## Node E

Пятая функция проверяет преобразование вида:

```text
y = (7 * x) mod 251
```

Используемые 8 байт в стеке:

```text
b1 3b 55 2d 7a 00 00 00
```

Отсюда восстанавливается последний блок:

```text
at0r}
```

и дальше еще три нулевых байта, потому что фактический флаг короче 40 байт, а буфер был нулен.

То есть последний осмысленный кусок:

```text
at0r}
```

## Сборка флага

Склеиваем 5 блоков:

```text
THEM?!CT
F{n0rm4L
_g0r0ut1
n3_val1d
at0r}
```

И получаем:

```text
THEM?!CTF{n0rm4L_g0r0ut1n3_val1dat0r}
```

## Проверка

При запуске:

```text
wine checker 'THEM?!CTF{n0rm4L_g0r0ut1n3_val1dat0r}'
```

бинарь отвечает:

```text
Correct!
```
